## P.2 Environment Variables & Configuration

### Why Environment Variables?

```python
# BAD: Hardcoded values
API_KEY = "sk-12345-secret-key"  # Leaked if committed!
DATABASE_URL = "localhost:5432"   # Breaks in production!

# GOOD: Environment variables
API_KEY = os.getenv("API_KEY")           # Read from environment
DATABASE_URL = os.getenv("DATABASE_URL") # Different per environment
```

### The Twelve-Factor App

Industry standard for cloud-native applications:

| Factor | Description | Our Implementation |
|--------|-------------|-------------------|
| Codebase | One repo, many deploys | Git |
| Dependencies | Explicitly declared | requirements.txt |
| **Config** | **Store in environment** | **.env, config.py** |
| Backing services | Treat as attached resources | Environment URLs |
| Build, release, run | Separate stages | CI/CD pipeline |
| Processes | Stateless | API is stateless |
| Port binding | Export via port | API_PORT env var |
| Concurrency | Scale via processes | API_WORKERS |
| Disposability | Fast startup/shutdown | Docker |
| Dev/prod parity | Keep similar | Docker |
| **Logs** | **Treat as streams** | **logger.py** |
| Admin processes | One-off processes | Makefile |

## P.7 Security Considerations

### Security Issues Found and Fixed

| Issue | Risk | Solution |
|-------|------|----------|
| Hardcoded secrets | High | Use .env files |
| CORS allow all | Medium | Configure specific origins |
| Error message exposure | Low | Sanitize error responses |
| No input validation | High | Pydantic validation (done) |
| Unsafe deserialization | Medium | Verify model integrity |

### Security Best Practices

```python
# 1. NEVER commit secrets
API_KEY = os.getenv("API_KEY")  # Not hardcoded!

# 2. Validate all inputs (Pydantic does this)
class IrisFeatures(BaseModel):
    sepal_length: float = Field(ge=0, le=10)  # Bounded!

# 3. Sanitize error messages in production
if settings.is_production:
    return {"error": "An error occurred"}  # Generic
else:
    return {"error": str(exception)}  # Detailed for debugging

# 4. Use HTTPS in production
# Configured in Kubernetes/load balancer
```

## 1.3 Dataset: Iris Classification

### What is the Iris Dataset?

The Iris dataset is one of the most famous datasets in machine learning, introduced by statistician Ronald Fisher in 1936.

```
┌─────────────────────────────────────────────────────────┐
│                    IRIS FLOWER                          │
│                                                         │
│        Sepal                    Petal                   │
│     ┌─────────┐              ┌─────────┐               │
│     │ length  │              │ length  │               │
│     │  width  │              │  width  │               │
│     └─────────┘              └─────────┘               │
│                                                         │
│  We measure these 4 features to predict the species    │
└─────────────────────────────────────────────────────────┘
```

### Dataset Summary

| Property | Value |
|----------|-------|
| Total Samples | 150 |
| Features | 4 (all numeric) |
| Classes | 3 (Setosa, Versicolor, Virginica) |
| Samples per Class | 50 (balanced) |
| Missing Values | None |

### The 4 Features

```python
features = [
    'sepal length (cm)',  # Length of the sepal
    'sepal width (cm)',   # Width of the sepal
    'petal length (cm)',  # Length of the petal
    'petal width (cm)'    # Width of the petal
]
```

### The 3 Classes (Target Variable)

```
Class 0: Setosa      → Easily separable
Class 1: Versicolor  → Some overlap with Virginica
Class 2: Virginica   → Some overlap with Versicolor
```

## 1.4 Exploratory Data Analysis (EDA)

### What is EDA?

EDA is the process of analyzing datasets to summarize their main characteristics, often using visual methods.

```
┌─────────────────────────────────────────────────────────┐
│                    EDA WORKFLOW                         │
│                                                         │
│  1. Load Data → 2. Check Quality → 3. Statistics →     │
│  4. Visualize → 5. Find Patterns → 6. Document         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Why is EDA Important?

| Reason | Explanation |
|--------|-------------|
| Data Quality | Find missing values, outliers, errors |
| Understanding | Learn feature distributions and relationships |
| Feature Selection | Identify which features matter most |
| Model Selection | Understand if data is linearly separable |
| Preprocessing | Determine if scaling/normalization needed |

### Key EDA Steps in Our Script (`src/ml/eda.py`)

#### Step 1: Load Data
```python
from sklearn.datasets import load_iris
import pandas as pd

iris = load_iris()
df = pd.DataFrame(data=iris.data, columns=iris.feature_names)
df['target'] = iris.target
```

**What's happening?**
- `load_iris()` - Loads built-in Iris dataset from sklearn
- `pd.DataFrame()` - Converts numpy array to pandas DataFrame
- DataFrames are easier to manipulate and analyze

#### Step 2: Check Data Quality
```python
# Check for missing values
df.isnull().sum()

# Check data types
df.dtypes

# Check shape
df.shape  # (150, 6)
```

**Key Questions:**
- Any missing values? → No
- Correct data types? → All numeric ✓
- Expected size? → 150 samples ✓

#### Step 3: Statistical Analysis
```python
df.describe()  # Shows: count, mean, std, min, 25%, 50%, 75%, max
```

**Example Output:**

**What This Tells Us:**
- Features have different scales (might need normalization)

- Petal features have higher variance

#### Step 4: Class Distribution

**Why It Matters:**
- **Balanced classes** = Model won't be biased
- **Imbalanced classes** = Need techniques like SMOTE, undersampling

#### Step 5: Correlation Analysis
```python

```

```
┌─────────────────────────────────────────────────────────────┐
│                    FULL DATASET (150 samples)               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────┬───────────────┐   │
│  │         TRAINING SET (80%)          │  TEST SET     │   │
│  │           120 samples               │  (20%)        │   │
│  │                                     │  30 samples   │   │
│  │   Used to TRAIN the model           │  Used to      │   │
│  │   Model learns patterns here        │  EVALUATE     │   │
│  │                                     │  performance  │   │
│  └─────────────────────────────────────┴───────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Code Explanation

```python
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(
    X, y,
    test_size=0.2,      # 20% for testing
    random_state=42,    # Reproducibility
    stratify=y          # Maintain class balance
)
```

| Parameter | Value | Meaning |
|-----------|-------|---------|
| `test_size` | 0.2 | 20% of data for testing |
| `random_state` | 42 | Same split every run (reproducible) |
| `stratify` | y | Each class has same % in train and test |

### Why Stratify?

Without stratification, random split might give:
- Training: 45 setosa, 50 versicolor, 25 virginica (unbalanced!)
- Test: 5 setosa, 0 versicolor, 25 virginica (no versicolor!)

With stratification:
- Training: 40 setosa, 40 versicolor, 40 virginica (balanced)
- Test: 10 setosa, 10 versicolor, 10 virginica (balanced)

## 2.4 Preprocessing with StandardScaler

### Why Scale Features?

```
Before Scaling:                  After Scaling:
─────────────────               ─────────────────
sepal_length: 4.3 - 7.9         sepal_length: -1.87 to 2.49
sepal_width:  2.0 - 4.4         sepal_width:  -2.43 to 3.09
petal_length: 1.0 - 6.9         petal_length: -1.56 to 1.78
petal_width:  0.1 - 2.5         petal_width:  -1.44 to 1.71

Features have different        All features have mean=0
ranges!                        and std=1
```

### The Formula

```
z = (x - mean) / std

Example:
If sepal_length = 5.0, mean = 5.84, std = 0.83
z = (5.0 - 5.84) / 0.83 = -1.01
```

### Why Does Scaling Matter?

| Algorithm | Needs Scaling? | Why |
|-----------|----------------|-----|
| Logistic Regression | Yes | Gradient descent converges faster |
| SVM | Yes | Distance-based, sensitive to scale |
| Decision Tree | No | Split based on thresholds |
| Random Forest | No | Ensemble of decision trees |
| KNN | Yes | Distance-based |

## 2.5 scikit-learn Pipelines

### What is a Pipeline?

A Pipeline chains preprocessing and model into one object:

```python
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression

pipeline = Pipeline([
    ('scaler', StandardScaler()),      # Step 1: Scale
    ('classifier', LogisticRegression()) # Step 2: Model
])

# Now fit and predict work on the whole pipeline
pipeline.fit(X_train, y_train)
predictions = pipeline.predict(X_test)
```

### Why Use Pipelines?

```
WITHOUT Pipeline (BAD):
───────────────────────
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # Fit on training
X_test_scaled = scaler.transform(X_test)        # Transform test
model.fit(X_train_scaled, y_train)
# Easy to make mistakes! What if you fit scaler on test data?

WITH Pipeline (GOOD):
─────────────────────
pipeline.fit(X_train, y_train)   # Handles everything
pipeline.predict(X_test)          # Automatically scales first
# No data leakage possible!
```

## 2.7 Evaluation Metrics Deep Dive

### Confusion Matrix

```
                        PREDICTED
                 Setosa  Versicolor  Virginica
              ┌────────┬───────────┬──────────┐
     Setosa   │   10   │     0     │    0     │  ← All correct!
ACTUAL        ├────────┼───────────┼──────────┤
   Versicolor │    0   │     9     │    1     │  ← 1 wrong
              ├────────┼───────────┼──────────┤
   Virginica  │    0   │     0     │   10     │  ← All correct!
              └────────┴───────────┴──────────┘

Diagonal = Correct predictions (29/30 = 96.7% accuracy)
Off-diagonal = Errors (1 versicolor predicted as virginica)
```

### Understanding Precision, Recall, F1

```
Example: Binary classification (Spam Detection)

                 Predicted
              Spam    Not Spam
           ┌───────┬──────────┐
  Spam     │  80   │    20    │    TP=80, FN=20
Actual     ├───────┼──────────┤
  Not Spam │  10   │   890    │    FP=10, TN=890
           └───────┴──────────┘

PRECISION = TP / (TP + FP) = 80 / (80 + 10) = 88.9%
"Of emails I marked as spam, how many were actually spam?"

RECALL = TP / (TP + FN) = 80 / (80 + 20) = 80.0%
"Of all actual spam, how many did I catch?"

F1-SCORE = 2 * (P * R) / (P + R) = 2 * (0.889 * 0.80) / (1.689) = 84.2%
"Balanced measure of precision and recall"
```

### When to Use Which Metric?

| Scenario | Prioritize | Why |
|----------|------------|-----|
| Spam filter | Precision | Don't want to miss important emails |
| Cancer detection | Recall | Don't want to miss any cancer cases |
| Balanced importance | F1-Score | Good overall measure |
| Balanced classes | Accuracy | Simple and interpretable |

## 2.8 Cross-Validation

### What is Cross-Validation?

Instead of one train/test split, we do multiple:

```
┌─────────────────────────────────────────────────────────────┐
│                   5-FOLD CROSS-VALIDATION                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ Fold 1: [TEST] [Train] [Train] [Train] [Train] → Score: 95% │
│ Fold 2: [Train] [TEST] [Train] [Train] [Train] → Score: 97% │
│ Fold 3: [Train] [Train] [TEST] [Train] [Train] → Score: 93% │
│ Fold 4: [Train] [Train] [Train] [TEST] [Train] → Score: 96% │
│ Fold 5: [Train] [Train] [Train] [Train] [TEST] → Score: 94% │
│                                                             │
│ Final Score: 95% (+/- 1.5%)                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Why Use Cross-Validation?

- More reliable estimate of model performance
- Uses all data for both training and testing
- Shows variance in performance (stability)

```python

```

## Key Concepts Learned in Step 2

### 1. The Bias-Variance Tradeoff

```
                    Model Complexity →
                 Low ──────────────────▶ High

       High  │  ┌────────────────────────┐
             │  │                        │
      Error  │  │    ╲    Total Error ╱  │
             │  │     ╲             ╱    │
             │  │      ╲    ┌─────╱      │
             │  │       ╲   │   ╱        │
             │  │  Bias  ╲  │  ╱ Variance│
             │  │         ╲ │ ╱          │
       Low   │  │          ╲│╱           │
             │  └────────────────────────┘
                        Sweet spot
                     (optimal complexity)
```

- **High Bias (Underfitting)**: Model too simple
- **High Variance (Overfitting)**: Model too complex
- **Goal**: Find the balance (low total error)

### 2. Overfitting vs Underfitting

| Sign | Underfitting | Overfitting |
|------|--------------|-------------|
| Training accuracy | Low | High |
| Test accuracy | Low | Low |
| Solution | More complex model | Regularization, simpler model |

### 3. Model Selection Strategy

1. Start with simple models (Logistic Regression)
2. If underfitting → try more complex models
3. If overfitting → add regularization
4. Use cross-validation to compare fairly
5. Choose based on business requirements

## 3.7 Dependency Injection

### The Problem

```python
# BAD: Loading model on every request
@app.post("/predict")
async def predict(features: IrisFeatures):
    predictor = IrisPredictor()  # Slow! Loads model every time
    return predictor.predict(...)
```

### The Solution

```python
# GOOD: Load once, inject where needed
predictor = None

def get_predictor():
    global predictor
    if predictor is None:
        predictor = IrisPredictor()  # Load only once
    return predictor

@app.post("/predict")
async def predict(features: IrisFeatures):
    pred = get_predictor()  # Fast! Reuses loaded model
    return pred.predict(...)
```

### Code Example

```python
import requests

def predict(features: dict) -> dict:
    """Call the FastAPI backend."""
    try:
        response = requests.post(
            "http://localhost:8000/predict",
            json=features,
            timeout=10
        )

        if response.status_code == 200:
            return response.json()
        else:
            st.error(f"API Error: {response.status_code}")

    except requests.exceptions.ConnectionError:
        st.error("Cannot connect to API. Is it running?")

    return None
```
